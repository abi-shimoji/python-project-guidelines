# コミット単位ガイド

この文書は、Git のコミットをどの単位で分けるかのルールをまとめる。

コミットメッセージの形式は `COMMIT_MESSAGE_GUIDE.md` に従う。この文書では、メッセージ以前に「何を 1 コミットに含めるか」を扱う。

実際にコミット単位を整理する手順は、Cook Book の [コミット単位を整理したい](./cook-book/organize-commit-units.md) も参照する。

## 1. 基本方針

- 1 コミット 1 目的を守る
- 後から履歴を読んだときに変更意図が分かる単位にする
- revert、cherry-pick、レビューがしやすい粒度にする
- フォーマット変更、リファクタリング、挙動変更を混ぜない
- 一時的な `wip` コミットは共有前に整理する

履歴として読みにくい例:

```text
feat: update
fix: fix
wip
chore: misc changes
feat(api): add user search and refactor auth and format files
fix(api): fix user search
fix(api): fix user search again
fix(api): really fix user search
```

これらは、後から履歴を追うときに「何を、なぜ、どの範囲で変えたか」が分かりにくい。コミット履歴は作業ログではなく、変更意図を追跡するための記録として残す。

## 2. 良いコミット単位

良い例:

- バグの原因となる条件分岐を修正する
- その修正に対応するテストを追加する
- 1 つの API エンドポイントを追加する
- 1 つの設定ファイルを更新する
- ドキュメントの 1 テーマを更新する

判断基準:

- コミット単体で説明できる
- コミット単体で戻しても意味が通る
- コミット単体で `cherry-pick` しても破綻しにくい
- レビュー時に変更理由を追いやすい

## 3. 避けるコミット単位

避ける例:

- バグ修正、リファクタリング、フォーマット変更が混ざっている
- 複数機能の追加が 1 コミットに入っている
- 途中作業の `wip` がそのまま共有されている
- 動作に必要な変更が前後のコミットへ分散している
- 「ついで」の修正が含まれている

このようなコミットは、レビュー、revert、cherry-pick のどれでも扱いにくくなる。

## 4. 分け方の目安

### 4.1 機能追加

機能追加は、レビューしやすい境界で分ける。

例:

- データ構造やスキーマ変更
- 実装本体
- テスト
- ドキュメント

ただし、小さい変更であれば実装とテストを同じコミットに含めてもよい。重要なのは、コミット単体で意味が通ること。

### 4.2 バグ修正

バグ修正は、原因と修正範囲が 1 つに読める単位にする。

推奨:

- 再現テストと修正を同じコミットに含める
- 無関係なリファクタリングを混ぜない
- 修正理由がメッセージや本文から分かるようにする

### 4.3 リファクタリング

リファクタリングは挙動変更と分ける。

推奨:

- 名前変更、移動、構造整理を小さく分ける
- 挙動変更が必要な場合は別コミットにする
- 大きな機械的変更は、手作業の変更と混ぜない

### 4.4 フォーマット変更

フォーマット変更は原則として独立したコミットにする。

理由:

- 実質的な変更との差分が読みやすくなる
- blame やレビューでノイズを減らせる
- revert 時に意図しない変更を巻き込みにくい

## 5. コミット前の確認

コミット前に差分を確認する。

```bash
git status
git diff
git add -p
git diff --cached
```

確認内容:

- 1 つの目的だけが含まれているか
- 不要なファイルや生成物が入っていないか
- テストやドキュメントの追加漏れがないか
- `cherry-pick` しても成立する単位になっているか

## 6. rebase によるコミット単位の整理

共有前の作業ブランチでは、`rebase -i` を使ってコミット単位を整理できる。

> [!WARNING]
> 共有済みブランチの履歴を `rebase -i` で書き換える場合は、影響範囲を確認する。
> 他人が利用しているブランチでは原則として避ける。

ベースブランチからのコミットを整理する:

```bash
git fetch origin
git rebase -i origin/main
```

直近 3 コミットだけ整理する:

```bash
git rebase -i HEAD~3
```

よく使う操作:

- `pick`: そのまま残す
- `reword`: コミットメッセージだけ修正する
- `edit`: コミット内容を修正する
- `squash`: 前のコミットへまとめ、メッセージも編集する
- `fixup`: 前のコミットへまとめ、メッセージは捨てる
- `drop`: コミットを削除する

## 7. rebase でよく行う整理

### 7.1 細かすぎるコミットをまとめる

途中作業のコミットは、意味のある単位にまとめる。

例:

```text
pick 1111111 feat(api): add user search endpoint
fixup 2222222 fix typo
fixup 3333333 adjust test
```

### 7.2 大きすぎるコミットを分ける

`edit` で対象コミットに止まり、差分を分け直す。

```bash
git reset HEAD^
git add -p
git commit -m "feat(api): add user search endpoint"
git add -p
git commit -m "test(api): cover user search endpoint"
git rebase --continue
```

### 7.3 メッセージを修正する

コミット単位は正しいがメッセージだけ直す場合は `reword` を使う。

```text
reword 1111111 feat(api): add user search endpoint
```

## 8. cherry-pick を前提にした注意

`cherry-pick` される可能性がある変更は、特にコミット単位を厳密にする。

- コミット単体でビルドやテストが通る見込みがある
- 前後のコミットに暗黙依存していない
- 取り込み先に不要な変更を含んでいない
- リリースブランチへ反映しても問題ない範囲に限定されている

コミット単位が粗い場合は、`cherry-pick` する前に元ブランチ側でコミットを整理する。

## 9. アンチパターン一覧

### 9.1 何でも入りコミット

特徴:

- 機能追加、バグ修正、リファクタリング、フォーマット変更が 1 コミットに混ざっている
- 差分量が大きく、レビュー時に主目的が分かりにくい

対処:

- 目的ごとに `git add -p` で分ける
- フォーマット変更は独立したコミットにする

### 9.2 WIP 放置コミット

特徴:

- `wip`, `tmp`, `fix later` のような途中作業のコミットがそのまま残っている
- 後から履歴を読んでも変更意図が分からない

対処:

- 共有前に `git rebase -i` で `fixup` または `squash` する
- 残す場合は意味のあるコミットメッセージに `reword` する

### 9.3 依存分散コミット

特徴:

- あるコミット単体ではビルドやテストが通らない
- 必要な変更が前後のコミットに分散している
- `cherry-pick` すると不足変更が発生する

対処:

- コミット単体で成立するように差分を組み直す
- 依存する変更は同じコミットに含めるか、明確な順序で分ける

### 9.4 後追い修正の連続

特徴:

- `fix typo`, `fix test`, `fix again` のような補修コミットが連続している
- 最終的な変更意図より、作業過程だけが履歴に残っている

対処:

- 共有前に対象コミットへ `fixup` する
- 意味のある変更単位だけが履歴に残るよう整理する

### 9.5 ついで修正コミット

特徴:

- 本来の目的とは関係ない軽微な修正が混ざっている
- レビュー対象や revert 対象が広がる

対処:

- ついでの修正は別コミットに分ける
- 今回の変更に不要なら別ブランチや別 PR に回す

### 9.6 巨大リファクタリングコミット

特徴:

- 名前変更、移動、構造変更、挙動変更がまとめて入っている
- 差分が大きく、レビューや不具合調査で原因を追いにくい

対処:

- 機械的な移動や名前変更を先に独立させる
- 挙動変更は別コミットにする
- 大きな構造変更は段階的に分ける

### 9.7 メッセージだけ整っているコミット

特徴:

- Conventional Commits の形式には合っているが、差分の中身が複数目的になっている
- `feat`, `fix`, `refactor` の境界が実際の変更内容と一致していない

対処:

- メッセージではなく差分の目的を基準に分ける
- 1 コミット 1 目的になっているか `git diff --cached` で確認する
