# 公開サーバーセキュリティガイドライン

このガイドラインは、Web サーバー、API サーバー、メールサーバーなど、インターネットから到達可能なサーバーを構築・運用する際の基本方針を定める。

具体的なコマンドは [Security Cook Book](./cook-book/README.md) を参照する。クラウド事業者、OS、ミドルウェアの仕様や、契約・社内規程により厳しい基準がある場合は、その基準を優先する。

過去の事例と本ガイドラインの対応は、[公開サーバー運用から学ぶ重要サイバー攻撃事例](./incident-cases/IMPORTANT_CYBER_INCIDENTS.md) を参照する。

## 1. 適用範囲

- グローバル IP アドレスを持つサーバー
- ロードバランサー、CDN、WAF などを経由して公開されるサーバー
- インターネットから接続可能な管理画面、VPN、踏み台サーバー
- クラウド上の仮想マシン、コンテナ、マネージドサービス

検証用や一時利用のサーバーも、インターネットから到達可能であれば本ガイドラインの対象とする。

## 2. 基本原則

- 公開するサービス、ポート、接続元を必要最小限にする
- OS、ミドルウェア、アプリケーションをサポート期間内のバージョンに保つ
- 管理アクセスと一般利用者のアクセスを分離する
- 単一の対策に依存せず、クラウド側とホスト側など複数の層で防御する
- 構築時だけでなく、監視、更新、バックアップ、廃止までを運用範囲とする
- サーバーの所有者、管理者、用途、公開ポート、連絡先を台帳で管理する

## 3. ネットワークとファイアウォール

### 3.1 公開ポート

受信通信は原則拒否とし、用途が説明できる通信だけを許可する。

| 用途 | 公開方針 |
| --- | --- |
| HTTPS | 一般公開する Web サービスでは許可する |
| HTTP | HTTPS への転送や証明書発行に必要な場合だけ許可する |
| SSH | 一般公開せず、VPN、踏み台、固定 IP の許可リストなどで接続元を制限する |
| データベース | 原則としてインターネットへ公開しない |
| 管理画面・監視・メトリクス | VPN や内部ネットワークからのみ接続可能にする |

- クラウドのセキュリティグループやネットワーク ACL と、OS のホストファイアウォールを併用する
- IPv4 と IPv6 の両方を確認し、片方だけが無制限に公開されないようにする
- ロードバランサーや CDN を使う場合、オリジンサーバーはそれらの接続元だけを許可する
- 大量通信への対策はホストファイアウォールだけに依存せず、クラウド事業者の DDoS 対策、CDN、レート制限を利用する
- WAF は補助的な防御として利用し、アプリケーションの脆弱性対策の代替としない
- 不要になった一時的な許可ルールは直ちに削除する
- 送信通信も用途を把握し、機密度や脅威に応じて接続先を制限する

ファイアウォールは、脆弱なサービスを安全にするものではない。公開中のサービス自体にも認証、更新、暗号化、入力検証などの対策を行う。

## 4. 管理アクセスと認証

- 管理者ごとに個別アカウントを発行し、共有アカウントを避ける
- SSH は公開鍵認証を使用し、鍵の動作確認後にパスワード認証を無効化する
- `root` の直接ログインを禁止し、必要な操作だけ `sudo` で昇格する
- 可能な場合は多要素認証を使用する
- 退職、異動、契約終了時は、アカウントと公開鍵を速やかに削除する
- 秘密鍵はパスフレーズで保護し、安全な端末または承認済みの鍵管理機能で管理する
- SSH のポート番号変更だけを主要な防御策としない

設定変更時は既存の管理セッションを維持し、別セッションで新しい設定による接続を確認してから切り替える。

## 5. OS とソフトウェア

- 不要なパッケージ、サービス、ユーザー、待受ポートを削除または無効化する
- セキュリティ更新を継続的に適用し、適用状況を監視する
- 自動更新を使用する場合は、対象パッケージ、再起動方針、メンテナンス時間、失敗通知を定める
- OS とミドルウェアのサポート終了日を管理し、期限前に移行する
- 外部リポジトリやインストールスクリプトは、提供元、署名、更新方法を確認してから利用する
- 本番変更は記録が残る方法で実施し、ロールバック手順を用意する

## 6. サービスとアプリケーション

- アプリケーションを `root` 権限で実行しない
- サービスごとに専用ユーザーを使用し、ファイルとディレクトリの権限を必要最小限にする
- データベース、キャッシュ、管理 API は内部ネットワークまたはローカルインターフェースで待ち受ける
- デバッグモード、初期アカウント、サンプルアプリケーション、ディレクトリ一覧表示を本番で無効化する
- エラーページやレスポンスから、スタックトレース、内部パス、製品の詳細バージョンを露出させない
- アップロード機能には、容量、拡張子、保存先、実行権限、マルウェア対策を設ける
- Web アプリケーションでは、認証、認可、入力検証、CSRF・XSS・SQL インジェクション対策を別途実施する

## 7. TLS と証明書

- Web サービスは HTTPS を使用し、HTTP は HTTPS へ転送する
- サポート中の TLS バージョンと暗号スイートを使用し、古いプロトコルを無効化する
- 証明書の有効期限を監視し、安全な方法で自動更新する
- 秘密鍵へのアクセス権を限定し、リポジトリやログへ保存しない
- リバースプロキシ、ロードバランサー、アプリケーション間の通信も、信頼境界に応じて暗号化する

## 8. 秘密情報とデータ

- パスワード、API キー、秘密鍵をソースコード、コンテナイメージ、起動スクリプトへ埋め込まない
- 承認されたシークレット管理機能を使用し、参照権限をサービス単位で分離する
- 秘密情報は定期的にローテーションし、漏えいの疑いがある場合は直ちに無効化する
- 保存データの機密度に応じて暗号化し、暗号鍵をデータと分離して管理する
- ログへ認証情報、セッション ID、個人情報を不用意に出力しない

## 9. ログ、監視、検知

- 認証、権限昇格、設定変更、ファイアウォール、アプリケーションエラーを記録する
- サーバー時刻を同期し、複数システムのログを時系列で追跡できるようにする
- ログをサーバー外へ転送し、侵害時に攻撃者が削除・改ざんしにくい状態にする
- ディスク使用率、CPU、メモリ、証明書期限、更新失敗、異常なログインを監視する
- アラートの通知先、対応時間、一次対応者を定め、定期的に通知試験を行う
- ログの保存期間とアクセス権を定める

## 10. バックアップと復旧

- 設定、データ、証明書、復旧に必要な秘密情報を対象にバックアップを設計する
- バックアップを本番サーバーと異なる障害範囲へ保存する
- バックアップへのアクセス権を限定し、必要に応じて暗号化する
- 取得成功だけでなく、定期的に復元試験を行う
- 復旧目標時間と復旧可能な時点を定める
- ランサムウェア対策として、変更・削除されにくい世代を保持する

## 11. 公開前チェック

- サーバーの所有者、用途、緊急連絡先が登録されている
- 公開ポートと接続元が設計どおりである
- IPv4 と IPv6 の両方で不要なポートが閉じている
- SSH の公開鍵認証と管理経路を別セッションで確認している
- OS、ミドルウェア、アプリケーションが更新済みである
- HTTPS、証明書更新、HTTP からの転送を確認している
- 初期パスワード、デバッグ設定、不要サービスを無効化している
- 秘密情報がリポジトリ、イメージ、ログに含まれていない
- ログ、監視、アラートの動作を確認している
- バックアップ取得と復元手順を確認している
- 障害時の切り戻し手順とインシデント連絡先が明確である

## 12. 運用と廃止

- 公開ポート、アカウント、鍵、パッケージ、証明書を定期的に棚卸しする
- 脆弱性情報とサポート期限を継続的に確認する
- 一時サーバーには終了日を設定し、利用後は放置せず廃止する
- 廃止時は DNS、証明書、ファイアウォールルール、アカウント、秘密情報を削除する
- ディスクやスナップショットは、データ保持方針に従って安全に消去する
